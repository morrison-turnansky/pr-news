# PR Review Filter

## Purpose

AI-assisted PR review filter that detects critical bugs using Claude Code agent via Google Vertex AI. Designed to be repository-agnostic with optional domain-specific skills.

**Design Philosophy**: Precision over recall — minimize false positives, default to PASS.

**Code Guidelines**:

TDD: All functions require a corresponding test. Python tests use Pytest. Follow Red-Green-Refactor phases.

Code Style:

Python: ruff  · pyrefly type checker · Google-style docstrings (no types in docstring body) · module-level docstrings on every .py file · docstrings on every test function · functions <200 lines (prefer <100)

Avoid local imports
Avoid try except. this is okay for claude call and web api calls. otherwise we want the error thrown
Avoid verbose inline comments
Use explicit module imports (e.g., `from pr_filter.config import load_config`) rather than importing from package __init__

**Schema-Prompt Alignment**: When modifying data structures or prompts, verify schema-prompt consistency by running `python3 -m pytest tests/test_schema_prompt_alignment.py -v`. The prompt must match the JSON schema generated by Pydantic, especially for the Verdict field (integer 0/1, not strings).

## Key Components

### Dependency Tree (pr_filter/)

Shows import relationships between modules (what depends on what):

```
data_structs.py (foundation - no internal dependencies)
├─ config.py
│  └─ claude_runner.py
│     └─ critique.py
├─ prompts.py
│  └─ critique.py
├─ review_parser.py
│  └─ critique.py
├─ filter.py (standalone - only imported by entry point)
└─ output.py (standalone - only imported by entry point)
```

**Key insights:**
- `data_structs.py` is the foundation with no internal dependencies
- `critique.py` is the integration point that depends on config, prompts, claude_runner, and review_parser
- `filter.py` and `output.py` only depend on data_structs and are called directly by the entry point

### Execution Flow (pr_filter/)

Shows the order files are called during a typical PR review run:

**1. Load Configuration**
- **config.py** - `load_config()` reads JSON config file
  - Returns `PRReviewConfig` with repository, skill paths, filter criteria, Vertex AI settings

**2. Fetch PRs**
- **filter.py** - `fetch_prs()` queries GitHub API
  - Filters by label, author, date range, merge status, open status, draft status
  - Populates files_changed from diff parsing
  - Returns list of `PullRequest` objects with diffs

**2.5. Diff Filter**
- **filter.py** - `diff_filter()` filters by diff characteristics
  - max_lines_changed: Filter out PRs exceeding line threshold
  - max_files_changed: Filter out PRs exceeding file threshold
  - only_test_files: Filter by test file status (all test / has non-test / all)
  - Returns filtered list of PRs

**3. Review Each PR** (loop - fresh agent per PR)

For each PR, the flow is:

- **3a. prompts.py** - `build_review_prompt()` generates prompt
  - Constructs review prompt with PR diff and skill file paths
  - `get_review_json_schema()` generates JSON schema for validation
  - Skills referenced by path (Claude uses Read tool if needed)

- **3b. critique.py** - `critique_pr()` orchestrates the review
  - Calls prompts.py to build prompt and schema
  - Calls claude_runner.py to execute Claude
  - Calls review_parser.py to parse output
  - Returns `ReviewResult`

- **3c. claude_runner.py** - `run_claude_code()` executes Claude CLI
  - Runs `claude -p --output-format json --json-schema`
  - Uses Vertex AI configuration from PRReviewConfig
  - Returns raw JSON output

- **3d. review_parser.py** - Parses Claude output into structured data
  - `parse_review_comments()` - Convert JSON to ReviewComment objects
  - `parse_review_summary()` - Convert JSON to ReviewSummary with defensive defaults
  - `determine_overall_verdict()` - Compute final verdict (BLOCK or PASS)
  - `create_error_result()` - Generate fallback on failure

- **3e. output.py** - Display and export results
  - `print_review()` - Human-readable structured review output
  - `format_review()` - Formatted text with comments, summary, verdict

**4. Export Results**
- **output.py** - `export_json()` writes all reviews to JSON file

**Data Models** (used throughout):
- **data_structs.py** - All Pydantic models
  - `PRFilter`, `PullRequest`, `ReviewComment`, `ReviewSummary`, `ReviewResult`
  - `ReviewOutputSchema`, `PRReviewConfig`, `ModelParams`
  - `Verdict` (BLOCK/PASS), `Confidence` (HIGH/MEDIUM)

- **end_to_end.py** - End-to-end demo showing complete workflow: fetch PRs → analyze with Claude → export results

Load with: `config = load_config("config.json")`

## Review Process

1. **Fetch** - Query GitHub API for PRs matching filter (label, author, date, merge/open/draft status)
2. **Critique** - For each PR:
   - Build prompt with diff + optional domain-specific skills
   - Execute `claude -p --output-format json --json-schema <schema>`
   - Parse structured JSON output (comments + summary)
   - Determine verdict: BLOCK (critical/major bugs) or PASS
   - Display formatted review to console
3. **Export** - Write all review results to JSON file

## Design Decisions

### Fresh Agent Per PR (No Session Reuse)
Each PR is reviewed by a fresh Claude Code agent (`-p` print mode). No conversation history is shared between PRs to avoid bias or context leakage.

### Structured Output with JSON Schema
Uses `--output-format json --json-schema` to force Claude Code to return structured JSON matching Pydantic schema. No markdown fences or conversational text.

### Pydantic for All Data Models
All data structures use Pydantic BaseModel for:
- Automatic JSON validation and parsing
- Type safety with runtime checks
- Automatic JSON schema generation for Claude Code
- Single source of truth for data structure definitions

No conversion layer needed between validation schemas and runtime models.

**CRITICAL: Verdict Field Constraints**

The `Verdict` field uses `IntEnum` and **MUST** remain an integer type:
- `Verdict.BLOCK = 0` (integer value 0)
- `Verdict.PASS = 1` (integer value 1)

**DO NOT:**
- Change `Verdict` to a string enum
- Add field validators that accept string values ("BLOCK", "PASS")
- Use `Verdict | str` type union

**Reason:** The prompt instructs Claude to output integer values (0, 1) for the verdict field. The JSON schema generated from Pydantic expects `{"type": "integer", "enum": [0, 1]}`. Any mismatch between the prompt and schema causes `error_max_structured_output_retries` because Claude's output fails schema validation.

**When modifying:**
1. If you change `data_structs.py` verdict type → update `prompts.py` examples to match
2. If you change `prompts.py` verdict format → update `data_structs.py` type to match
3. Run `python3 -m pytest tests/test_schema_prompt_alignment.py -v` to verify consistency
4. Update all test files to use the correct verdict format
